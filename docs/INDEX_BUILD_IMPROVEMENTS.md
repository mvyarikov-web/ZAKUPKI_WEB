# Улучшения процесса построения индекса

**Дата:** 18 октября 2025 г.

## Проблемы

1. **Долгое построение индекса без обратной связи**: При индексации большого количества файлов пользователь не видел прогресса, не было ясно, что происходит.

2. **Ошибка JSON парсинга при таймауте**: При превышении таймаута запроса middleware возвращал текст вместо JSON, что приводило к ошибке: `Unexpected token 'З', "Запрос пре"... is not valid JSON`.

## Решения

### 1. Улучшенное логирование прогресса индексации

**Файл:** `document_processor/search/indexer.py`

Добавлена двухпроходная индексация:
- **Первый проход**: подсчёт общего количества файлов
- **Второй проход**: индексация с логированием прогресса

**Изменения:**
- Логирование общего количества файлов в начале
- Прогресс каждые 10 файлов: `"Прогресс индексации: 50/200 (25%)"`
- DEBUG-лог для каждого файла с номером и именем
- Итоговое сообщение с общим количеством проиндексированных файлов

**Пример логов:**
```
INFO: Подсчёт файлов для индексации...
INFO: Найдено файлов для индексации: 156
INFO: Прогресс индексации: 10/156 (6%)
INFO: Прогресс индексации: 20/156 (12%)
DEBUG: Индексация [23/156]: contracts/Договор_2024.pdf
...
INFO: Прогресс индексации: 156/156 (100%)
INFO: Индексация завершена: uploads/_search_index.txt (156 файлов)
```

### 2. JSON-ответы от middleware при таймауте

**Файл:** `webapp/utils/timeout_middleware.py`

**Было:** При таймауте или ошибке возвращался plain text
```python
return [f"Запрос превышает лимит времени {self.timeout} сек.".encode('utf-8')]
```

**Стало:** Корректный JSON-ответ
```python
error_json = json.dumps({
    'success': False,
    'error': f'Запрос превышает лимит времени {self.timeout} сек. Попробуйте позже.'
}, ensure_ascii=False)
start_response('504 Gateway Timeout', [('Content-Type', 'application/json; charset=utf-8')])
return [error_json.encode('utf-8')]
```

### 3. Увеличен таймаут по умолчанию

**Файл:** `webapp/config.py`

Таймаут увеличен с 30 до 120 секунд для поддержки индексации больших архивов:
```python
REQUEST_TIMEOUT = int(os.environ.get('REQUEST_TIMEOUT', '120'))
```

### 4. Улучшенная визуализация на фронтенде

**Файл:** `static/js/script.js`

**Добавлена функция безопасного парсинга JSON:**
```javascript
function safeFetchJson(response) {
    if (!response.ok) {
        return response.json().catch(() => {
            throw new Error(`HTTP ${response.status}: ${response.statusText}`);
        }).then(data => {
            throw new Error(data.error || data.message || `Ошибка ${response.status}`);
        });
    }
    return response.json();
}
```

**Улучшен индикатор прогресса `rebuildIndexWithProgress()`:**
- Более информативное сообщение: "Построение индекса… (это может занять некоторое время)"
- Отображение времени выполнения: "Готово за 45.3с"
- Обработка ошибок с визуальной индикацией (красная полоса прогресса)
- Модальное окно с детальным сообщением об ошибке
- Увеличенное время отображения результата (2 сек вместо 0.6 сек)

**Применение во всех fetch-запросах:**
Все вызовы `.then(res => res.json())` заменены на `.then(safeFetchJson)` для единообразной обработки ошибок.

## Результат

1. ✅ Пользователь видит прогресс индексации в логах приложения (`logs/app.log`)
2. ✅ Нет ошибок парсинга JSON при таймаутах
3. ✅ Увеличенный таймаут позволяет индексировать большие архивы
4. ✅ Улучшенная визуальная обратная связь на фронтенде
5. ✅ Корректная обработка всех типов ошибок (сеть, таймаут, HTTP-статусы)

## Тестирование

Для проверки изменений:

1. Загрузите папку с большим количеством файлов (100+)
2. Наблюдайте прогресс в `logs/app.log`:
   ```bash
   tail -f logs/app.log
   ```
3. При превышении таймаута увидите корректное JSON-сообщение вместо ошибки парсинга
4. В UI индикатор прогресса покажет информативное сообщение и время выполнения

## Переменные окружения

Для изменения таймаута установите переменную окружения:
```bash
export REQUEST_TIMEOUT=180  # 3 минуты
python3 app.py
```
