# Инкремент 13 — Оптимизация OCR и двухэтапная индексация

## Постановка

На основе анализа текущей реализации выявлены три критические проблемы, замедляющие работу пользователя:

1. **Медленное распознавание OCR** — текущая реализация обрабатывает до 4 ориентаций для каждой страницы, делая полное распознавание 4 раза (0°, 90°, 180°, 270°). Для документа на 10 страниц это 40 OCR-операций вместо 10.

2. **Низкое качество распознавания** — используется универсальный режим OCR без настройки под специфику рус/англ текстов; нет предобработки изображений (контраст, шум, бинаризация).

3. **Блокирующая индексация** — пользователь не может работать с текстовыми документами, пока система распознаёт сканы. Весь процесс индексации выполняется синхронно за один запрос с таймаутом 120 секунд.

**Текущие узкие места (из анализа кода):**

```python
# reader.py: _auto_orient_image() — 4 полных OCR на страницу
for angle in [0, 90, 180, 270]:
    text = pytesseract.image_to_string(cropped, lang=lang, config='--psm 6')
    # Анализ качества и выбор лучшей ориентации
```

```python
# indexer.py: create_index() — синхронная обработка всех файлов
for rel_path, abs_path, source in self._iter_sources(root_folder):
    text, meta = self._extract_text(abs_path, rel_path, source)  # Блокирующий OCR
    self._write_entry(out, rel_path=source, text=text, meta=meta)
```

```python
# reader.py: _extract_text_ocr() — без предобработки изображений
images = convert_from_path(path)
for img in images:
    img_corrected = self._auto_orient_image(img, lang)  # 4x OCR
    t = pytesseract.image_to_string(img_corrected, lang=lang)  # generic config
```

**Цели инкремента:**
- Ускорить OCR в 4-8 раз за счёт оптимизации ориентации и предобработки
- Повысить качество распознавания рус/англ текстов на 15-30%
- Обеспечить доступность текстовых файлов для поиска в течение 10-30 сек
- Фоновая обработка OCR с инкрементальным обновлением индекса

Ограничения:
- Не ломать существующий функционал (загрузка, поиск, удаление)
- Graceful degrade при отсутствии tesseract/poppler
- Работа в рамках текущего таймаута 120 секунд для первого этапа

## Спецификация

### Анализ текущей архитектуры

**Процесс индексации (синхронный):**
1. POST `/build_index` → `DocumentProcessor().create_search_index(uploads/)`
2. `Indexer.create_index()` — два прохода:
   - Первый: подсчёт файлов
   - Второй: извлечение текста из ВСЕХ файлов последовательно
3. Запись в `uploads/_search_index.txt`, затем перемещение в `index/_search_index.txt`
4. Обновление `file_statuses` с char_count для каждого файла

**Обработка PDF (текущая):**
- `PdfReader.read_pdf()` → анализ типа PDF
- Векторный: каскад экстракторов (pdfplumber → pypdf → pdfminer → pymupdf)
- Скан: `_extract_text_ocr()` → автоориентация для каждой страницы

**Узкие места OCR:**
1. **Автоориентация (4x overhead):**
   ```python
   # Для КАЖДОЙ страницы:
   for angle in [0, 90, 180, 270]:
       rotated = img.rotate(angle)
       cropped = rotated.crop(box)  # 50% центральная область
       text = pytesseract.image_to_string(cropped, lang=lang, config='--psm 6')
       # Вычисление score и выбор лучшей
   ```
   
2. **Отсутствие предобработки:**
   - Нет бинаризации, удаления шума, улучшения контраста
   - Используется generic PSM (6) вместо оптимизированного для документов
   
3. **Лимиты:**
   - `max_pages_ocr=2` — обрабатывается max 2 страницы на документ
   - `budget_seconds=5` — жёсткий таймаут на векторное извлечение
   
4. **Отсутствие кэша ориентации:**
   - Каждая страница анализируется независимо, хотя обычно документ имеет единую ориентацию

### Функциональные требования

**FR-001 Двухэтапная индексация:**
- Этап 1 (FAST): Индексация файлов без OCR (TXT, DOCX, XLSX, векторные PDF, HTML и т.д.)
- Этап 2 (SLOW): OCR-обработка сканированных PDF
- Частичный индекс после Этапа 1 доступен для поиска

**FR-002 Оптимизация автоориентации:**
- Использовать OSD (Orientation and Script Detection) Tesseract вместо 4x полного OCR
- Кэшировать ориентацию на уровне документа (если первая страница 0° → остальные тоже)
- Fallback на упрощённую эвристику при недоступности OSD

**FR-003 Предобработка изображений:**
- Бинаризация (Otsu thresholding) для повышения контраста
- Удаление шума (median filter)
- Resize до оптимального DPI (300) если изображение слишком большое/маленькое

**FR-004 Оптимизация настроек Tesseract:**
- PSM 6 (uniform text block) для документов вместо generic
- Белый список символов: кириллица + латиница + цифры + пунктуация
- Отключение словарей для ускорения (только необходимые для рус+англ)

**FR-005 Лимиты и производительность:**
- Целевая скорость: 1-2 страницы/сек с OCR (вместо текущих 0.2-0.3)
- Увеличить `max_pages_ocr` с 2 до 10 (при сохранении разумного времени)
- Адаптивный тайм-бюджет для OCR этапа

**FR-006 Прогресс и статусы:**
- Отдельные счётчики для Этапа 1 и Этапа 2
- UI отображает: "Индексация текста: 50/100", "OCR: 5/15 (в процессе)"
- Логи с детализацией по этапам и скорости обработки

**FR-007 Инкрементальное обновление индекса:**
- После Этапа 1: запись в `index/_search_index.txt` (частичный)
- После каждого OCR-файла: дозапись в конец индекса (append mode)
- Поиск работает по растущему индексу

**FR-008 Устойчивость:**
- Этап 1 работает даже без tesseract/poppler
- Ошибки OCR не блокируют весь процесс
- Пропуск недоступных/повреждённых файлов с логированием

**FR-009 Совместимость:**
- Сохранить формат `_search_index.txt`
- Не ломать `/search`, `/view`, `/upload`, `/delete`
- Обратная совместимость с `FilesState` и светофорами

**FR-010 Конфигурация:**
- Новые параметры в `config.py`:
  ```python
  OCR_USE_OSD = True  # использовать OSD для ориентации
  OCR_CACHE_ORIENTATION = True  # кэшировать ориентацию документа
  OCR_PREPROCESS = True  # предобработка изображений
  OCR_DPI_TARGET = 300  # целевой DPI
  OCR_WHITELIST = 'rus_eng_digits'  # белый список символов
  OCR_MAX_PAGES = 10  # увеличено с 2
  INDEX_STAGE_1_TIMEOUT = 60  # таймаут для этапа 1
  INDEX_STAGE_2_TIMEOUT = 300  # таймаут для этапа 2 (5 мин)
  ```

### Пользовательские сценарии

**Сценарий 1: Смешанная папка (текст + сканы)**
1. Пользователь загружает папку: 80 DOCX + 20 PDF (10 векторных, 10 сканов)
2. Нажимает "Построить индекс"
3. Через 20-30 сек: "Этап 1 завершён: 90 файлов. Поиск доступен."
4. Через 1-2 мин: "Этап 2 завершён: +10 файлов с OCR. Индекс полный."
5. Пользователь может искать с момента окончания Этапа 1

**Сценарий 2: Только сканы**
1. Пользователь загружает 50 PDF-сканов
2. Нажимает "Построить индекс"
3. Этап 1: ~5 сек, индекс пуст, UI показывает "OCR в процессе: 0/50"
4. OCR обрабатывает по 1-2 стр/сек, обновляет прогресс каждые 5-10 файлов
5. Через 3-5 мин: "Индекс готов: 50 файлов с OCR"

**Сценарий 3: Ошибки и fallback**
1. Tesseract недоступен → Этап 1 работает, Этап 2 пропускается с предупреждением
2. Повреждённый PDF → логируется, пропускается, не блокирует остальные
3. Таймаут на Этапе 2 → частичный индекс сохраняется, ошибка в UI

### Ключевые сущности

**IndexingJob** (новая сущность):
- `job_id`: UUID задачи индексации
- `stage`: 1 (text) или 2 (ocr)
- `total_files`: общее количество
- `processed_files`: обработано
- `start_time`, `end_time`: временные метки
- `status`: running, completed, error
- `errors`: список ошибок

**DocumentMetadata** (расширение):
- `requires_ocr`: bool — флаг необходимости OCR
- `ocr_pages`: int — количество страниц для OCR
- `orientation`: int — кэшированная ориентация (0/90/180/270)
- `preprocessing_applied`: bool — применялась ли предобработка

def _preprocess_image(self, img) -> Any:
    """Бинаризация, шумоподавление, DPI нормализация"""
    import cv2
    import numpy as np
    
    # Конверсия в grayscale
    gray = cv2.cvtColor(np.array(img), cv2.COLOR_RGB2GRAY)
    
    # Otsu binarization
    _, binary = cv2.threshold(gray, 0, 255, cv2.THRESH_BINARY + cv2.THRESH_OTSU)
    
    # Median filter для шумоподавления
    denoised = cv2.medianBlur(binary, 3)
    
    return Image.fromarray(denoised)

**5.1 Параллельная обработка OCR**
```python
import concurrent.futures

def process_ocr_batch(self, ocr_files: list, max_workers: int = 2):
    """Параллельная обработка OCR файлов"""
    with concurrent.futures.ThreadPoolExecutor(max_workers=max_workers) as executor:
        futures = {executor.submit(self._extract_ocr_file, f): f for f in ocr_files}
        for future in concurrent.futures.as_completed(futures):
            result = future.result()
            self._append_to_index(result)  # атомарная дозапись
```

### Дата начала: 18 октября 2025 г.

### Что сделано

**1. Двухэтапная индексация (FR-001, частично)**
- Создан класс `TwoStageIndexer` для разделения файлов на текстовые и требующие OCR
- Классификация выполняется перед индексацией
- Этап 1 обрабатывает TXT, DOCX, XLSX, векторные PDF
- Этап 2 откладывается для OCR-файлов (пока работает синхронно)

**2. Real-time прогресс индексации (FR-006, полностью)**
- Внедрена архитектура Server-Sent Events (SSE) для потоковой передачи прогресса
- Индексация запускается в фоновом потоке (`threading.Thread`)
- Endpoint `/build_index` возвращается немедленно
- Endpoint `/build_index_progress` передаёт обновления каждые 300 мс
- JavaScript подключается к SSE и обновляет UI в реальном времени
- Пользователь видит:
  - Счётчики обработанных/всего файлов (обновляются по мере обработки)
  - Имя текущего обрабатываемого файла
  - Прогресс-бар заполняется постепенно, не моментально
- Создан `IndexingProgressManager` для потокобезопасного хранения состояния

**3. Единый прогресс-индикатор (FR-006, полностью)**
- UI отображает единую шкалу прогресса для обоих этапов
- Обновления происходят файл-за-файлом, а не одним блоком в конце
- После завершения показывается итоговое сообщение с количеством файлов и временем

**4. Тестирование**
- Минимальный тест с таймаутом 10 секунд (`test_realtime_progress.py`)
- Проверяет вызовы `progress_callback` при индексации
- Выполнение: 0.07 секунды ✓

### Метрики текущей реализации

**UX:**
- ✅ Прогресс отображается в реальном времени (файл за файлом)
- ✅ Немедленный ответ от сервера (< 100 мс)
- ✅ Понятная обратная связь (имя файла, счётчики, прогресс-бар)
- 🔄 Поиск доступен только после полной индексации (пока нет append mode)

**Производительность:**
- ✅ Индексация не блокирует HTTP-запросы
- ✅ Фоновый поток с daemon=True (завершается при остановке сервера)
- 🔄 OCR не оптимизирован (пока 4x overhead на ориентацию)

**Надёжность:**
- ✅ Потокобезопасный `IndexingProgressManager`
- ✅ Graceful завершение SSE при completed/error
- ✅ Контекст Flask в фоновом потоке через `app_context()`
